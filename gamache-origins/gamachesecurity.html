Security and Integrity

This page defines the security posture, provenance posture, and canonical-record expectations for the Gamache Origins branch materials. This is an integrity page, not a magic shield. Public static files can be copied, mirrored, or altered elsewhere; the point here is to identify what is official, what is non-canonical, and how integrity should be evaluated.

Security Posture

This branch is intentionally built as a lightweight static record surface with no JavaScript dependency, no form processing, no inline user input handling, and a restrictive content security posture.

  • No client-side script is required for normal operation.
  • No user-account access, authentication, or form submission is performed on these pages.
  • No data collection workflow is built into the ancestry pages themselves.
  • All internal page navigation is static and file-based.
  • Content is intended to be human-readable, reference-oriented, and provenance-conscious.

Tamper-Evident Principle

This branch is not “tamper-proof” in the literal sense. Public web files can be copied, altered, republished, quoted out of context, or mirrored elsewhere by third parties. The correct security concept here is tamper-evident integrity, not fantasy invulnerability.

  • Only the canonical repository branch and canonical folder path should be treated as authoritative.
  • Copies, screenshots, mirrors, forks, reposts, or extracted fragments should not automatically be treated as official.
  • Where content conflicts arise, the maintained canonical branch version should control.
  • Future hash manifests may be used to strengthen integrity verification.

Integrity Controls Already in Use

Current Hardening Features

  • Static HTML architecture with no required JavaScript runtime.
  • Restrictive Content Security Policy delivered through the page itself.
  • form-action 'none' to prevent form submission behavior.
  • frame-ancestors 'none' to reduce unauthorized framing.
  • connect-src 'none' to prevent network callouts from the page.
  • object-src 'none' and media-src 'none' to narrow the attack surface.
  • No dependence on external stylesheets, fonts, or third-party scripts.

Recommended Next Integrity Step

Hash Manifest

The next serious step for integrity is to publish a hash manifest for the canonical files. That would let readers compare file hashes and detect whether files were altered after publication.

A later file such as SHA256SUMS.txt or a branch-specific integrity manifest would strengthen the project materially. That is real integrity practice, unlike pretending an HTML filename alone is armor.
